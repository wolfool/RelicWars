# RelicWars AI Development Guidelines

이 문서는 RelicWars 마인크래프트 플러그인 개발을 주도하는 AI를 위한 핵심 아키텍처 가이드라인 및 마일스톤(작업 분할) 문서입니다. 개발 중 이 문서를 나침반으로 삼아 기능을 구현합니다.

## 1. 플러그인 아키텍처 및 기술 스택 (How to Build)

### 1.1 기술 스택
- **API:** Paper API 1.21.4 (NMS 사용을 지양하고 Paper 고유 API 적극 활용)
- **언어:** Java 21 이상
- **빌드 툴:** Maven (`pom.xml`)
- **데이터베이스:** SQLite (유저 데이터, 팀 정보 저장) + YAML (설정값 저장)
- **데이터 영속성 보장:** `PersistentDataContainer` (PDC)를 활용해 유물 아이템의 고유 번호, 쿨타임, 바닥에 떨어진 봉인 유물의 방치 시간 등을 엔티티 레벨에 저장. (서버 재시작 대비)

### 1.2 코어 매니저 구조 (Core Managers)
플러그인은 다음과 같은 책임(Manager) 기반 구조로 나뉘어 설계됩니다.
- **`RelicManager`:** 유물 아이템 생성, NBT/PDC 태그 부여, 우클릭 액티브 능력 처리, 쿨타임 관리.
- **`CombatManager`:** 다운(Downed) 상태 전이, 2초 무적, 5타 확킬 판정, 자동 드랍(가장 낮은/높은 번호 계산) 로직 처리.
- **`TeamManager`:** 팀 결성(최대 2인), 아군 오인 사격(FF) 방지, 팀 구조(부활) 로직, 최대 10개 보유 엔딩 조건 추적.
- **`BossManager`:** 바닐라 몹 기반의 커스텀 보스 스폰(속성/체력/패턴 커스텀), 보스 스킬 스케줄러, 처치 시 봉인 유물 드랍 처리.
- **`EventManager`:** 30분 단위 방치 유물 소문 에스컬레이션, 블러드문(야간 확률 이벤트) 좌표 방송 시스템, 최종 보스(#000) 소환 관리.

## 2. 개발 마일스톤 (Task Breakdown)

개발은 방대한 코드가 엉키지 않도록 아래의 페이즈(Phase) 단위로 쪼개어 단계적으로 진행됩니다. 하나의 페이즈가 완벽히 검증되면 다음 페이즈로 넘어갑니다.

### Phase 1: 기초 뼈대 구축 (Project Scaffolding)
- Maven 프로젝트 설정 (`pom.xml` 의존성 주입)
- `plugin.yml` 작성 (플러그인 정보, 명령어, 권한 등록)
- 메인 클래스(`RelicWars.java`) 및 기초 매니저 싱글톤 등록
- SQLite DB 연결 및 `config.yml` 로드 시스템 구현

### Phase 2: 유물 기초 및 커스텀 아이템 시스템
- `RelicManager` 기초 구현
- PDC(PersistentDataContainer)를 활용한 커스텀 아이템 생성 유틸리티 구현
- 우클릭 이벤트 감지 및 아이템 NBT 쿨타임 추적 시스템(주사용 손 강제 포함) 구현
- 아이템 보관 제한(엔더 상자, 셜커 상자 등 블록 상호작용 캔슬) 구현

### Phase 3: 다운 및 교전(Combat) 시스템
- `CombatManager` 구현 (EntityDamageEvent 제어)
- 체력 0 도달 시 다운(기절) 상태 전이 및 2초 무적 로직
- 다운 시 가장 낮은 번호 유물 바닥 드랍 (ItemDisplay 엔티티 활용한 '봉인 유물' 구현)
- 5타 평타 확킬 판정 및 확킬/자결 시 유물 비례 드랍 로직 구현
- 다운된 팀원 구조(우클릭 8초) 및 3칸 체력(6 HP) 부활 구현

### Phase 4: 이벤트 및 시스템 룰 구현
- `EventManager` 구현
- 방치된 봉인 유물의 타이머 추적 및 소문 방송 (30/60/90/120분 단위 Y좌표 포함)
- 블러드문 확률 발생 및 오버월드/네더 타겟 좌표 차원 방송
- 전투 중 강제 종료(Combat Logging) 감지 및 사망 처리 로직

### Phase 5: 커스텀 보스 및 엔딩 시스템
- `BossManager` 구현
- Paper API의 Attribute, Entity AI (혹은 스케줄러 패턴)를 활용한 보스 제작 기반 마련
- 모든 유물 스폰 여부 추적 및 '한 팀 10개 이상 보유' 엔딩 조건 개방 로직 구현
- 중앙 제단 30분 방어(King of the Hill) 타이머 로직 구현

### Phase 6: 31개 유물 능력 개별 구현
- 가장 작업량이 많은 구간으로, 유물들을 5개 단위(Batch)로 끊어서 순차적으로 구현 및 테스트 진행.
- 낙뢰, 텔레포트, 투명화, 무적, 위치 추적 등 다양한 액티브/패시브 스킬 코딩.

### Phase 7: 마무리 및 최적화
- 권한 및 관리자 명령어(`/relic give`, `/relic locate` 등) 구현
- 전체 메모리 누수 점검 및 PDC 데이터 최적화
- 최종 테스트 및 1.0.0 릴리즈 준비
